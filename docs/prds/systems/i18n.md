## Summary

`@pie-qti/i18n` is the framework's translation and locale-formatting system. It provides a thin, type-safe layer over browser-native `Intl` APIs that enables QTI player components to display localized strings without requiring host applications to configure anything. Components accept an optional `i18n?: I18nProvider` prop; when it is absent they fall back to hardcoded English strings via the `??` operator. Locale switching persists via `localStorage` and propagates by triggering a page reload — a deliberate simplification made possible by the low frequency of locale changes in an assessment session.

---

## Background and rationale

### Why a custom package rather than i18next, Fluent, or similar

The framework's bundle budget is strict: the entire player including the default locale must gzip to under 10 KB. Every general-purpose i18n library brings a pluralization engine, a message-format parser, SSR utilities, and plugin infrastructure that together exceed this budget before a single translation key is added. The framework's requirements are narrower: dot-notation key lookup, `{variable}` interpolation, `one`/`other` pluralization, and number/date formatting. All of these are handled by a ~2 KB core plus `Intl.NumberFormat` and `Intl.DateTimeFormat`, which are zero-cost because they ship in every modern browser. A custom package gives full control over the public surface without pulling in unused code.

### Why i18n is an optional prop, not a required context

Assessment items are deployed in many environments: standalone demo pages, LMS iframes, native mobile webviews, internal authoring tools. Many of these environments never display non-English text and have no interest in wiring up an i18n provider. Making i18n optional means components work out of the box in English with no host configuration, and only hosts that actually need localization need to provide a provider. The `??` fallback pattern (e.g. `i18n?.t('common.submit') ?? 'Submit'`) keeps each component self-sufficient; it cannot silently render an empty string or a raw key.

### Why locale changes trigger a page reload

Components are compiled as Svelte custom elements (web components). Standard Svelte context and stores do not cross Shadow DOM boundaries; the host document's context is invisible inside a custom element's shadow root. The alternatives — a global singleton observable, a custom event bus, a shared `localStorage` listener with `StorageEvent` — all require either a global mutable object or event wiring that must be set up correctly in every host environment. Given that locale changes in an assessment session are rare (typically only at session start, not mid-test), a page reload is acceptable UX and eliminates the entire reactive-propagation problem. The chosen locale is written to `localStorage` so that `DefaultI18nProvider` reads it on the next page load without requiring any additional host coordination.

### Why en-US is loaded eagerly and other locales lazily

English is the fallback locale for every missing key in every other locale. If the en-US bundle is not present when a component first renders, the fallback chain breaks and components display raw keys instead of English text. Loading it eagerly (via Vite's `import.meta.glob` with `{ eager: true }`) guarantees it is always available. Additional locales add ~8 KB gzipped each; loading them lazily via `loadLocale()` avoids penalizing English-only deployments.

### Why partial locale coverage is acceptable

Translation coverage for non-English locales is intentionally incomplete (es-ES ~80%, fr-FR ~70%, nl-NL/ro-RO/th-TH ~30%). Missing keys fall through the lookup chain to the en-US fallback rather than surfacing an error or a raw key. This is preferable to blocking a locale deployment until 100% coverage is reached: a partially-translated UI is better than no translated UI, and the fallback to English is unambiguous and readable. The `bun run check-translations` tool tracks coverage precisely so product teams can prioritize translator work based on actual key counts rather than estimates.

## Functional requirements

- **FR-1:** Components must render a functional English UI when no `i18n` prop is provided.
- **FR-2:** When an `i18n` prop is provided, `t(key)` must return the translation for the current locale, or the fallback string, or the en-US value — in that priority order — never a raw key or an empty string.
- **FR-3:** `setLocale(locale)` on `SvelteI18nProvider` must store the locale code in `localStorage` under the key `pie-qti-locale` and then call `window.location.reload()`.
- **FR-4:** `DefaultI18nProvider` must read `pie-qti-locale` from `localStorage` on construction and use it as the active locale, overriding the constructor's `locale` parameter.
- **FR-5:** `loadLocale(locale)` must fetch the locale module asynchronously and cache it; a second call for the same locale must return immediately without a network request.
- **FR-6:** Custom messages passed to `DefaultI18nProvider(locale, customMessages)` must take priority over framework translations for matching keys, while missing keys still fall through to the framework translations.
- **FR-7:** `addCustomMessages(locale, messages)` must deep-merge the provided messages into any previously set custom messages for that locale.
- **FR-8:** `t(key, defaultString)` must return `defaultString` when the key is not found in any locale, rather than returning the key itself.
- **FR-9:** `t(key, { variable: value })` must replace `{variable}` occurrences in the resolved string with the provided value.
- **FR-10:** `plural(key, { count })` must resolve to `${key}.one` when `count === 1` and `${key}.other` otherwise, then apply interpolation.
- **FR-11:** `formatNumber(value, options?)` and `formatDate(date, options?)` must delegate to `Intl.NumberFormat` and `Intl.DateTimeFormat` respectively, using the current locale.
- **FR-12:** `bun run check-translations` must enumerate all keys from the en-US locale, report which keys are missing in each non-English locale, and exit non-zero if any keys are missing.
- **FR-13:** `bun run scan-hardcoded` must scan `.svelte` and `.ts` files under a target path and report lines containing string literals that match known translation values, with the suggested `i18n?.t(key) ?? 'fallback'` replacement.
- **FR-14:** `bun run build` must run the translation coverage check before compilation; builds must fail if the check exits non-zero (unless `PIE_QTI_DEV_FAST_BUILD=1` is set).
- **FR-15:** All message keys exported from `@pie-qti/i18n` must be TypeScript-typed as `MessageKey`, derived from the structure of `en-US.ts`, so that invalid keys produce a compile-time error.

## Non-functional requirements

- **Accessibility:** The `LocaleSwitcher` reference component must have an accessible `<label>` associated with the `<select>` via `for`/`id`, and the `<select>` must carry an `aria-label` from `i18n.selectLanguageAriaLabel`. All translatable ARIA labels (region labels, live-region announcements, screen-reader-only text) must be in the `accessibility.*` and interaction-specific namespaces; hardcoded ARIA text in components is not acceptable.
- **Performance:** Core i18n logic must gzip to ≤2 KB. en-US locale must gzip to ≤8 KB. Each additional locale must gzip to ≤8 KB. Total default bundle (core + en-US) must not exceed 10 KB gzipped. Locale modules must be code-split so that loading a second locale does not block initial render.
- **Cross-platform:** `localStorage` access must be guarded with `typeof localStorage !== 'undefined'` checks. `window.location.reload()` must be guarded with `typeof window !== 'undefined'`. These guards allow the package to be imported in SSR or Node.js test environments without throwing.
- **Security:** No user-supplied content is ever evaluated. Interpolation replaces `{variable}` tokens with string-coerced values; there is no `eval`, `Function()`, or HTML-rendering step in the translation pipeline. Custom messages are treated as trusted data supplied by the host application at initialization time, not as user input.
- **i18n:** The package is itself the i18n layer; no external i18n dependency is permitted. BCP 47 locale codes must be used throughout (`en-US`, not `en_US` or `en`). The `LocaleCode` type is open (`string & {}`) to allow host applications to register locales not shipped with the framework.

## Design decisions

### Page-reload locale switching

**Decision:** `SvelteI18nProvider.setLocale()` writes the locale to `localStorage` and then calls `window.location.reload()`.

**Rationale:** Svelte custom elements render inside Shadow DOM. Svelte's context API and store subscriptions do not cross the Shadow DOM boundary, so a standard reactive store cannot propagate a locale change from the host page into a mounted web component. The alternatives — a global singleton observable registered on `window`, a `StorageEvent` listener, a custom DOM event bus — each require explicit setup in the host and create coupling between the framework and the host's JavaScript environment. Locale changes during an active assessment session are operationally rare (typically the user selects a locale before starting, not mid-test). A page reload is therefore low-cost in practice and eliminates an entire class of synchronization bugs.

**Alternatives considered:**
- Global reactive store on `window` (`window.__pieQtiI18n`): works across Shadow DOM but creates an implicit global dependency and makes testing harder.
- `StorageEvent` listener: asynchronous; components would need to subscribe and re-render, which reintroduces the Shadow DOM propagation problem.
- CSS custom properties for locale state: only applicable to static string selection, not useful for dynamic key lookup.

**Consequences:** Locale switching always causes a full page reload. This is a non-issue in assessment delivery (locales are set once), but makes locale switching visually abrupt in authoring tools. Host applications that need smooth locale switching must implement their own mechanism and pass a fresh `I18nProvider` instance to components after the transition.

---

### Optional i18n prop with `??` fallback

**Decision:** Every component declares `i18n?: I18nProvider` as an optional prop. String access uses `i18n?.t('key') ?? 'English fallback'` at the call site.

**Rationale:** Requiring a provider forces every host (including internal tooling, unit tests, and Storybook stories) to configure i18n even when they only ever render English. The optional pattern means the package adds zero friction for English-only deployments. The explicit `??` fallback guarantees that a missing provider never produces an empty string or a thrown error — the fallback is always a readable English string hardcoded at the call site.

**Alternatives considered:**
- Required prop with a default value (`i18n = new DefaultI18nProvider()`): creates a new provider instance per component mount; provider construction reads `localStorage`, which is a side effect in a Svelte `$props()` default.
- Module-level singleton provider: hidden global state that makes components non-portable and hard to test.
- Svelte context injection: does not cross Shadow DOM boundaries (see above).

**Consequences:** Each component that uses i18n must maintain its own list of English fallback strings. If a translation key is renamed, the fallback strings are not automatically updated. The `bun run scan-hardcoded` tool exists specifically to catch drift between fallback strings and the en-US locale.

---

### Custom package instead of i18next

**Decision:** `@pie-qti/i18n` is a bespoke package with a ~2 KB core.

**Rationale:** The framework needs: key lookup with dot notation, `{var}` interpolation, `one`/`other` pluralization, and `Intl`-based number/date formatting. i18next at minimum adds ~9 KB gzipped for the core alone, before plugins for interpolation or `Intl` integration. The bespoke package stays within the 10 KB total budget. There is no server-side rendering requirement; no need for ICU MessageFormat; no need for namespace lazy-loading beyond what Vite's `import.meta.glob` provides.

**Alternatives considered:**
- i18next: too large; plugin ecosystem adds complexity without providing needed features.
- Fluent (Mozilla): runtime is ~12 KB gzipped; FTL syntax is non-trivial to author; no advantage over `Intl` for the pluralization rules needed.
- LinguiJS: requires a compile step and a catalog format that does not fit the framework's TypeScript-native locale files.

**Consequences:** The package does not support ICU MessageFormat syntax (e.g. `{count, plural, one {...} other {...}}`). Complex pluralization rules beyond `one`/`other` (needed for Slavic languages with multiple plural forms) are not natively supported. Hosts that need them must provide a custom `I18nProvider` implementation. The `transform.` namespace in en-US contains some ICU-style strings (e.g. `'{count, plural, one {# item} other {# items}}'`) that are passed through as-is and would need a host-side ICU parser to render correctly — this is a known gap.

---

### Locale coverage strategy

**Decision:** en-US is the single source of truth for the key schema. Other locales are partially translated and fall back to en-US for missing keys.

**Rationale:** Requiring 100% coverage before shipping a locale would block localization work indefinitely, since translation is often done incrementally by contractors. The fallback-to-English behavior is unambiguous: users see English text for untranslated strings rather than a key or empty space. The `check-translations` build gate prevents regressions (a key present in one build cannot silently disappear in the next), while the partial-coverage policy allows forward progress.

**Alternatives considered:**
- Block shipping until 100% translated: infeasible given contractor translation timelines.
- Display `[MISSING: key]` for untranslated strings: visible to end users in K-12 assessments; unacceptable UX.
- Crowdin/Phrase integration for automated coverage tracking: adds external dependency; current volume does not justify it.

**Consequences:** Users of partially-translated locales will see English strings for missing keys without any visual indicator that a translation is absent. The `check-translations` report must be reviewed by the i18n maintainer before each release to track coverage trends and prioritize translator work.

---

### TypeScript key inference from en-US

**Decision:** `MessageKey` is derived from the shape of `en-US.ts` via a `NestedKeyOfHelper` utility type. The locale file uses `as const` to enable deep literal inference.

**Rationale:** Maintaining a separate type definition for message keys would immediately become stale. Deriving the type from the live locale file means that adding a key to en-US automatically makes it available (and required) in the TypeScript type, and removing a key causes compile errors at all call sites. This catches translation key renames and deletions at build time rather than at runtime.

**Alternatives considered:**
- Manually maintained string union: would immediately drift from the actual locale file.
- Code generation (`codegen` scripts): adds a build step; the TypeScript inference approach achieves the same result without generated files.

**Consequences:** `MessageKey` only covers keys present in en-US. Keys in other locales that do not exist in en-US are invisible to TypeScript. This is by design: en-US is the canonical schema; extra keys in other locales are treated as translator mistakes and flagged as "obsolete" by `check-translations`.

## Extension points

| Extension point | Interface/type | How to use | Example |
|----------------|---------------|------------|---------|
| Custom i18n provider | `I18nProvider` | Implement all required methods; pass instance as `i18n` prop | Wrap i18next and delegate `t()`, `formatNumber()`, etc. |
| Custom locale messages | `Record<string, LocaleMessages>` constructor parameter | Pass to `DefaultI18nProvider(locale, customMessages)` or call `addCustomMessages()` post-construction | Override `common.submit` with brand-specific copy |
| Additional locale files | Any BCP 47 code + `LocaleMessages` | Register via `addCustomMessages('de-DE', messages)` | Add German translations not shipped by the framework |
| Locale persistence key | Hardcoded `'pie-qti-locale'` in `localStorage` | Read this key at app startup to pre-set the locale before mounting components | SSR hydration or test environment setup |
| `LocaleSwitcher` UI | Reference component in `src/components/` | Copy and adapt; not exported from the package's main entry point | Host-branded locale picker |

## Data model / contracts

Key types are defined in `packages/i18n/src/core/I18nProvider.ts` and `packages/i18n/src/core/types.ts`.

### `I18nProvider` interface (invariants not obvious from types)

- `t()` must never return `undefined` or throw. When a key is missing at every level of the fallback chain, it returns either the explicit `defaultString` argument or the raw key string.
- `setLocale()` returns `Promise<void> | void`; callers must `await` it when the return value is a Promise, because `SvelteI18nProvider` uses the async form to load the locale before triggering the reload.
- `plural()`, `formatNumber()`, and `formatDate()` are optional on `I18nProvider`; callers must guard with `?.`.

### `DefaultI18nProvider` lookup chain

For a given key, the resolution order is:
1. `customMessages[currentLocale][key]`
2. `messages[currentLocale][key]` (framework translations)
3. `customMessages[fallbackLocale][key]` (en-US custom overrides)
4. `messages[fallbackLocale][key]` (en-US framework translations)

If all four return `undefined`, the explicit `defaultString` argument is returned; if that is also absent, the raw key is returned. No step throws.

### `LocaleCode` type

`FrameworkLocaleCode` is a closed union of eight BCP 47 codes shipped with the package. `LocaleCode` widens this to `FrameworkLocaleCode | (string & {})`, allowing any string while preserving autocomplete for framework codes. The `loadLocale()` method only accepts `FrameworkLocaleCode`; custom locales must be provided via `customMessages`.

### `MessageKey` type

Derived at compile time from the shape of `en-US.ts` using `NestedKeyOfHelper`. Expressed as dot-notation strings (e.g. `'interactions.choice.correct'`). Adding or removing a key in `en-US.ts` changes the type without any manual update.

### `localStorage` key

The persistence key is `'pie-qti-locale'`. Host applications that manage multiple `DefaultI18nProvider` instances (e.g., embedding multiple independent players on one page) share this key; the last `setLocale()` call wins after the page reloads.

## Acceptance criteria

### Functional

**AC-1: English fallback without provider**
```
Given: A component that uses i18n is mounted with no i18n prop
When: The component renders
Then: All user-visible strings display their hardcoded English fallback values (e.g., "Submit", "Cancel")
      and no string renders as an empty string or as a raw key like "common.submit"
```

**AC-2: Key lookup with active provider**
```
Given: A DefaultI18nProvider initialized with locale 'es-ES' with es-ES fully loaded
When: i18n.t('common.submit') is called
Then: The return value is "Enviar" (the Spanish translation)
```

**AC-3: Fallback to en-US for missing key**
```
Given: A DefaultI18nProvider initialized with locale 'nl-NL'
  and the nl-NL locale is loaded
  and the key 'interactions.hotspot.selected' is not present in nl-NL
When: i18n.t('interactions.hotspot.selected') is called
Then: The return value is "Selected:" (the en-US value)
      and no error is thrown or logged in production mode
```

**AC-4: Variable interpolation**
```
Given: A DefaultI18nProvider with en-US loaded
When: i18n.t('assessment.question', { current: 3, total: 10 }) is called
Then: The return value is "Question 3 of 10"
```

**AC-5: Explicit default string overrides key fallback**
```
Given: A DefaultI18nProvider with en-US loaded
When: i18n.t('nonexistent.key', 'My fallback') is called
Then: The return value is "My fallback", not "nonexistent.key"
```

**AC-6: Pluralization — singular**
```
Given: A DefaultI18nProvider with en-US loaded
When: i18n.plural('plurals.items', { count: 1 }) is called
Then: The return value is "1 item"
```

**AC-7: Pluralization — plural**
```
Given: A DefaultI18nProvider with en-US loaded
When: i18n.plural('plurals.items', { count: 5 }) is called
Then: The return value is "5 items"
```

**AC-8: Locale persistence and reload**
```
Given: A SvelteI18nProvider wrapping a DefaultI18nProvider
  and localStorage is available
When: setLocale('fr-FR') is called
Then: localStorage.getItem('pie-qti-locale') returns 'fr-FR' before the reload fires
  and window.location.reload() is called exactly once
```

**AC-9: Locale restored from localStorage on next load**
```
Given: localStorage contains 'pie-qti-locale' = 'es-ES'
When: new DefaultI18nProvider('en-US') is constructed
Then: provider.getLocale() returns 'es-ES', not 'en-US'
```

**AC-10: Custom messages take priority over framework translations**
```
Given: A DefaultI18nProvider('en-US', { 'en-US': { common: { submit: 'Send Answer' } } })
When: i18n.t('common.submit') is called
Then: The return value is "Send Answer", not "Submit"
```

**AC-11: addCustomMessages deep-merges without overwriting sibling keys**
```
Given: A DefaultI18nProvider('en-US') with no custom messages
When: addCustomMessages('en-US', { common: { submit: 'Send' } }) is called
  and i18n.t('common.cancel') is called
Then: The return value is "Cancel" (the framework default is preserved)
  and i18n.t('common.submit') returns "Send" (the custom override is applied)
```

**AC-12: loadLocale caches on second call**
```
Given: A DefaultI18nProvider with fr-FR not yet loaded
When: loadLocale('fr-FR') is called twice in sequence
Then: The second call completes synchronously (no additional module fetch)
  and both calls resolve without error
```

**AC-13: loadLocale for unknown locale throws**
```
Given: A DefaultI18nProvider
When: loadLocale('xx-XX') is called for a locale not in the framework bundle
Then: The promise rejects with an error message referencing the locale code
```

**AC-14: formatNumber uses active locale**
```
Given: A DefaultI18nProvider with locale 'fr-FR'
When: i18n.formatNumber(1234567.89) is called
Then: The return value uses French number formatting conventions (space as thousands separator, comma as decimal separator)
Notes: Exact output depends on browser Intl implementation; verify against Intl.NumberFormat('fr-FR').format(1234567.89)
```

**AC-15: Translation coverage check exits non-zero on missing keys**
```
Given: The check-translations script is run on a state where at least one non-English locale file is missing a key that exists in en-US
When: bun run check-translations runs
Then: The process exits with a non-zero exit code
  and the output names the affected locale and the missing key
```

**AC-16: build fails when translations are incomplete**
```
Given: A locale file is missing one or more keys relative to en-US
  and PIE_QTI_DEV_FAST_BUILD is not set
When: bun run build is executed
Then: The build exits non-zero before TypeScript compilation begins
```

**AC-17: TypeScript rejects invalid message key**
```
Given: A TypeScript file that calls i18n.t('nonexistent.key.path')
  where 'nonexistent.key.path' is not present in en-US.ts
When: TypeScript type checking runs (bunx tsc --noEmit)
Then: A type error is reported at the call site
```

### Edge cases

**AC-E1: Missing provider — no empty strings in ARIA attributes**
```
Given: A component is rendered with no i18n prop
When: The rendered DOM is inspected for aria-label, aria-describedby, and title attributes sourced from i18n calls
Then: None of these attributes contain an empty string; all fall back to their English hardcoded value
```

**AC-E2: localStorage unavailable (SSR/Node environment)**
```
Given: The DefaultI18nProvider constructor is called in an environment where localStorage is not defined
When: The constructor runs
Then: No ReferenceError is thrown; the provider initializes with the constructor's locale parameter
```

**AC-E3: window.location.reload unavailable (non-browser environment)**
```
Given: SvelteI18nProvider.setLocale() is called in a test environment where window is not defined
When: The method runs
Then: The locale is stored in localStorage (if available)
  and no ReferenceError is thrown from the window access
```

**AC-E4: setLocale for an unloaded locale logs a warning**
```
Given: A DefaultI18nProvider where 'de-DE' has not been loaded and no custom messages exist for 'de-DE'
When: setLocale('de-DE') is called directly (not via SvelteI18nProvider)
Then: A warning is logged to the console in development mode
  and the locale is set to 'de-DE' without throwing
  and subsequent t() calls fall back to en-US values
```

**AC-E5: Interpolation with missing variable leaves placeholder intact**
```
Given: A DefaultI18nProvider with en-US loaded
When: i18n.t('assessment.question', { current: 1 }) is called
  (note: 'total' variable is missing from the values object)
Then: The return value is "Question 1 of {total}" — the unresolved placeholder is preserved verbatim
Notes: This avoids silent data loss; the caller can see which variable was not supplied
```

**AC-E6: Check-translations reports obsolete keys**
```
Given: A non-English locale file contains a key that does not exist in en-US

  and the process exits non-zero
```

## Open questions

- [ ] **ICU plural strings in `transform.*` namespace**: Several keys (e.g. `transform.samples.itemCount`) use ICU MessageFormat syntax (`{count, plural, one {# item} other {# items}}`). These are passed through by `DefaultI18nProvider` without parsing. Should the framework add an optional ICU parser, or should these strings be rewritten to use the `one`/`other` key convention?
- [ ] **ar-SA and zh-CN locale files**: Both locale codes appear in `FrameworkLocaleCode` and have corresponding locale files in `src/locales/`, but they are not listed in the `README.md` as supported locales. What is their intended status — experimental, in-progress, or officially supported?
- [ ] **Shared `localStorage` key across multiple player instances**: If a host page embeds two independent item players, both share `pie-qti-locale`. A locale switch in one player triggers a reload that affects both. Should the key be nameable per-provider instance?
- [ ] **RTL support**: ar-SA is a right-to-left locale. The i18n package currently has no mechanism to expose directionality to components (e.g., a `getDirection()` method or a CSS custom property). This needs to be addressed before ar-SA can be released.
